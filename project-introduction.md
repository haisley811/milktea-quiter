# 今天不喝：项目完整介绍

`今天不喝` 是一个围绕奶茶、咖啡和甜饮习惯设计的移动端网页 App。它不是简单的记账工具，而是把“想喝但忍住了”和“今天真的喝了”都变成可记录、可反馈、可视化的小体验。

项目的核心想法是：用户不一定要靠严厉自律来戒奶茶，而是通过一次次轻量记录，看到自己省下的钱、减少的热量和糖分，以及角色体型变化带来的即时反馈。

## 一句话介绍

这是一个奶茶习惯管理 App：用户可以虚拟下单并选择“今天戒了”，也可以记录“今天喝了”；系统会用 AI 估算饮品的价格、热量和糖分，并用数据统计、Body Score 和会变化体型的小人持续反馈习惯变化。

## 适合谁用

1. 想少喝奶茶、果茶、咖啡但又不想被说教的人。
2. 想知道每次奶茶大概花了多少钱、摄入多少热量和糖分的人。
3. 喜欢轻量打卡、可爱角色反馈、数据可视化的人。
4. 想把“忍住没喝”也变成正向成就的人。

## 核心功能

### 1. 两种记录模式

App 设计了两个主要入口：

```text
今天戒了
今天喝了
```

`今天戒了` 是虚拟下单模式。用户像真的点奶茶一样选择饮品类型、甜度、杯型、配料等信息，但最后记录的是“我今天忍住了”。系统会把这次没喝换算成省下的钱、减少的热量和糖分。

`今天喝了` 是真实记录模式。用户记录自己今天实际喝掉的一杯饮品，系统会把它计入花费、热量、糖分和 Body Score 变化里。

这两个模式让用户不只看到“失败”，也能看到每一次没喝带来的具体收益。

### 2. 虚拟下单

虚拟下单是这个项目最有辨识度的功能之一。它保留了点奶茶时的选择过程：

1. 饮品类型：奶茶、果茶、咖啡等。
2. 杯型：中杯、大杯等。
3. 甜度：低糖、中糖、高糖等。
4. 配料：珍珠、椰果、奶盖等。
5. 饮品名称和备注。

用户可以像点单一样完成选择，但最后点击的是“今天戒了”。这会把冲动消费转化为一次“被记录的胜利”。

### 3. AI 估算价格、热量、糖分

项目接入了服务端 AI 估算能力。用户填写饮品信息后，系统会向后端接口发送饮品结构化信息，由 QWEN 模型估算：

1. 价格。
2. 热量。
3. 糖分克数。
4. 估算说明。

这个能力通过后端接口完成：

```text
/api/estimate-drink
```

前端和微信小程序不会直接接触 QWEN API Key，密钥只放在服务器环境变量里。

如果 AI 接口暂时不可用，项目会自动退回本地估算规则，保证用户仍然可以记录，不会因为 AI 失败而卡住。

### 4. 手动修正估算值

AI 估算后，用户仍然可以手动修改：

1. 价格。
2. 热量。
3. 糖分。

被用户改过的字段会显示 `已改` 标签。  
如果用户想恢复 AI 或本地建议值，也可以点击恢复建议值。

这个设计承认饮品差异很大：不同品牌、杯型和配方会导致结果不同，所以估算值不是强行固定，而是可以被用户校正。

### 5. Body Score

项目设计了一个 `Body Score`，范围是 0 到 100。

规则是：

1. 记录一次 `今天喝了`，Body Score 增加 2。
2. 记录一次 `今天戒了`，Body Score 减少 2。
3. 分数越低，角色越轻盈。
4. 分数越高，代表近期甜饮负担更重。

这个分数不是医学指标，而是行为反馈指标。它用一种直观但不严厉的方式告诉用户：最近的选择正在改变状态。

### 6. 会变化体型的小人

项目里有一组角色图片，会随着 Body Score 和阶段变化展示不同状态。角色不是静态装饰，而是习惯反馈的一部分。

角色状态包括：

1. 更轻盈的状态。
2. 健康活力的状态。
3. 有一点负担的状态。
4. 更需要调整的状态。

角色还准备了透明背景版本、眨眼版本和挥手版本，用于不同场景和微交互。

这种设计把抽象的“少喝一点”变成可见的身体反馈：用户不是只看数字，而是看到一个陪伴式角色在变化。

### 7. 首页统计

首页会汇总用户最关心的结果：

1. 省下的钱。
2. 减少的热量。
3. 减少的糖分。
4. Body Score。
5. 连续记录状态。
6. 今日记录概况。
7. 当前轻盈状态。

这些数据来自本地记录，会随着每次新增、删除或重置自动重新计算。

### 8. 数据页和历史记录

数据页用于回看长期变化。它包含：

1. 趋势和阶段反馈。
2. 当前状态说明。
3. 历史记录列表。
4. 按日期分组的记录。
5. 单条记录删除。
6. 清空所有记录。

删除记录后，项目不会只删页面文字，而是会重新计算所有统计数据和 Body Score，保证数据一致。

### 9. 本地保存

当前版本优先使用浏览器本地保存。固定存储 key 是：

```text
milkTeaAppData
```

这意味着：

1. 用户刷新页面后记录还在。
2. 不需要登录就能使用。
3. 数据保存在当前浏览器和设备里。
4. 更换设备或清理浏览器数据后，本地数据不会自动同步。

项目也准备了平台迁移说明，后续可以迁移到微信小程序存储、云数据库或 iOS 本地存储。

### 10. 个人页和迁移准备

个人页重点说明当前账号和同步状态：

1. 当前是本地体验账号。
2. 数据保存在本机。
3. 后续可接入微信小程序账号体系。
4. 后续可接入 iOS 或其他平台。

这让产品从第一版开始就为多端迁移留好了方向。

## 新颖点

### 1. 把“没喝奶茶”也产品化

多数饮食记录工具只记录已经吃了什么、喝了什么。这个项目反过来，把“我差点下单但忍住了”也变成一次完整记录。

它保留点单的满足感，又把结果转换成省钱、少糖、少热量的即时反馈。

### 2. 虚拟下单降低戒断阻力

用户不是被要求“不要想奶茶”，而是可以照样进入选择流程。选择完成后，系统告诉用户：如果这杯没有喝，你省下了什么。

这个设计更像温柔替代，而不是硬性禁止。

### 3. AI 和本地规则双轨估算

AI 能根据饮品名称、类型、甜度、杯型、配料等信息给出更灵活的估算。本地规则则保证离线或接口失败时仍然能继续使用。

所以体验不会因为 AI 波动中断。

### 4. Body Score 和角色形态绑定

项目没有只用冷冰冰的数字，而是让 Body Score 影响小人形态。用户每次选择都会让角色状态更轻盈或更有负担。

这是一种游戏化反馈，但语气保持轻柔，不制造羞耻感。

### 5. 手动修正让 AI 更可信

估算结果不是“系统说了算”。用户可以改价格、热量、糖分，并看到 `已改` 标签。  
这让 AI 结果更像辅助判断，而不是不透明裁决。

### 6. 为微信小程序提前设计接口边界

项目已经把 AI 能力放在服务端 route 里。这意味着微信小程序只需要请求网站后端，不需要暴露 QWEN API Key，也不需要在小程序端重写 AI 调用逻辑。

## UI 风格

整体 UI 是移动端 App 风格，不是传统桌面网站。

视觉关键词：

1. 轻盈。
2. 软糯。
3. 玻璃拟态。
4. 低压力。
5. 奶茶和健康之间的柔和对比。

主要风格包括：

1. 手机壳式页面宽度，适合 375px 到 430px 手机屏幕。
2. 半透明玻璃卡片。
3. 柔和的薄荷绿、粉色、紫色、琥珀色点缀。
4. 大面积留白和圆角面板。
5. 底部 Tab 导航。
6. 角色视觉位于首页核心区域。
7. 数据区使用紧凑但清晰的分组布局。

它不是健身 App 那种强压迫、强目标风，也不是普通记账软件的工具风，而是更像一个轻量陪伴式的日常习惯小应用。

### 可切换的 6 套视觉风格

当前版本已经加入全局 UI 风格切换。用户可以在 `我的` 页面点击按钮切换，切换后页面背景、手机外框、卡片、按钮、底部导航、输入框、弹窗、Toast、选中态和状态色都会跟随变化。

已提供的风格包括：

1. `软糖奶茶`：默认风格，粉紫、薄荷绿、玻璃卡片和柔和阴影。
2. `黑白极简`：黑白灰、高留白、细边框，适合想要更克制工具感的用户。
3. `液态玻璃`：通透蓝紫、强模糊、高光层，偏 iOS 液态玻璃质感。
4. `千禧甜心`：高饱和粉色、青色、黄色，带一点 Y2K 网页和贴纸感。
5. `新粗野`：粗黑边、硬阴影、高对比色块，像潮流活动页或海报式界面。
6. `霓虹赛博`：深色底、荧光绿蓝、高亮描边，更适合夜间和强个性视觉。

这些偏好会保存在浏览器本地，存储 key 是：

```text
milkTeaUiPreferences
```

### 显示或隐藏人物

`我的` 页面也加入了 `显示人物` 开关。  
打开时，首页、记录页、数据页会展示小人、Body Score、当前阶段和角色进度。  
关闭时，小人和人物相关 UI 部件会一起隐藏，包括首页角色图、记录页小角色、数据页角色图、Body Score 面板、阶段进度和提交结果里的 Body Score 反馈。

## 微交互特点

## 金币与换装系统

“今天不喝”把每一次克制也变成可累积的奖励。每成功戒掉 1 杯，用户获得 10 个金币；金币可在“我的”页面的角色衣橱中兑换服装。商店提供 10 件不同方向的潮流服装，从 10 杯即可兑换的女团舞台装，到 50 杯兑换的冰蓝暖冬大衣，价格对应坚持的长度，而不是抽奖或随机付费。

服装包括女团、汉服、街头丹宁、芭蕾、蝴蝶结、机能、吊带裙、通勤、未来偶像和冬装等方向。10 件服装均配有独立生成的透明背景预览插画。人物有 5 个会变化的 Body Score 身材阶段，服装随阶段使用 5 档比例适配，保持肩部、腰部和腿部位置稳定，不遮挡脸部和状态标签。购买、穿戴状态保存在 `milkTeaWardrobe`；清空记录时会同步清空金币和衣橱，保证奖励与历史数据一致。完整玩法见 `docs/wardrobe-and-coins.md`。

项目加入了很多小反馈，让操作更像真正的 App。

### 1. 点击涟漪

用户点击页面时，会出现 `tap-burst` 反馈。这让按钮和卡片不只是静态响应，而有轻微触感。

### 2. 触感反馈

项目封装了 `webHaptics`。在支持的设备上，重要操作会有轻微震动反馈。

### 3. 选择弹跳

饮品选项、模式卡片和按钮被选中时，有 `selected-pop` 效果。这让用户知道自己已经选中某项。

### 4. AI 估算状态

AI 估算时会显示：

1. 加载状态。
2. 状态点闪动。
3. 忙碌态按钮。
4. `AI 已连接` 或 `本地临时值` 这样的状态说明。

用户能知道当前结果来自 AI 还是本地兜底。

### 5. Toast 提示

项目有不同语气的 Toast：

1. 成功。
2. 提醒。
3. 警告。
4. 信息。

Toast 带有进度条和状态点，不会突然消失得很生硬。

### 6. 结果弹窗

提交后会出现结果弹窗，告诉用户这次记录带来的变化，例如省下多少钱、减少多少热量或增加了多少摄入。

### 7. 角色动效

角色区域有轻微浮动、眨眼和挥手资源准备。角色不是一个静止头像，而是更像陪伴反馈对象。

### 8. 表单聚焦

输入框聚焦时会有边框和背景变化。这让手机上填写饮品名称、价格、热量等信息时更清楚。

### 9. 删除和重置保护

删除单条记录后会重新计算数据，并给出成功提示。清空所有记录使用确认弹窗，避免误触。

## 技术和部署特点

虽然用户不需要懂代码，但项目内部已经做了这些准备：

1. 使用 Next.js App Router。
2. 使用 TypeScript。
3. 使用 Tailwind CSS。
4. 使用服务端 API route 保护 QWEN API Key。
5. 支持 `OPTIONS` 请求，便于微信小程序和跨端调用。
6. API 有请求体大小限制，避免异常大请求。
7. API 返回 no-store，避免估算结果被缓存。
8. 有 Netlify 部署配置。
9. 有 GitHub Actions 验证配置。
10. 有部署检查脚本。
11. 有公开 API 验证脚本。
12. 有微信小程序服务层模板。
13. 有面向 0 代码用户的双击式上线辅助工具。
14. 有 iOS 和微信迁移准备文档。

### 零代码上线与验收辅助

为了让没有代码基础的人也能按步骤上线，项目根目录准备了一组 Windows 双击脚本：

1. `launch-control.cmd`：上线控制台总入口，用数字菜单打开状态检查、GitHub 上传、Netlify 验收、微信模板检查、依赖安装、微信 AppID 设置、微信真机验收登记、项目介绍和排障手册。
2. `connect-github.cmd`：创建 GitHub 空仓库后，粘贴 HTTPS 仓库地址，自动配置 `origin` 并推送 `main`。
3. `after-netlify.cmd`：Netlify 部署成功后，粘贴公共链接，先验证公共 API，通过后自动更新微信小程序模板里的 `API_BASE_URL`，再运行微信模板检查和烟测，最后记录本地公共 API 验收状态。
4. `check-wechat-template.cmd`：检查微信小程序模板是否只请求 Netlify 后端、是否没有 `QWEN_API_KEY`、是否没有 QWEN 原始域名、`API_BASE_URL` 是否没有误带 `/api/estimate-drink`。
5. `set-wechat-api-url.cmd`：已单独验证公共 API 后，只更新微信小程序模板的后端根域名。
6. `set-wechat-appid.cmd`：正式上传微信小程序前，把模板里的 `touristappid` 改成你的真实 `wx...` 小程序 AppID。
7. `mark-wechat-verified.cmd`：微信后台 request 合法域名保存成功、真机预览请求成功后，把验收结果写入本地 `.launch-state.json`。
8. `check-npm-network.cmd`：只检查本机 npm registry 网络，不安装依赖，帮助判断 `package-lock.json` 生成失败是不是网络问题。
9. `install-local-deps.cmd`：本机网络正常时安装依赖、运行验证并执行生产构建；如果本机 npm 继续超时，可以继续让 Netlify 云端安装依赖。
10. `verify-public-api.cmd`：只验证线上公共 API 是否能完成 GET、OPTIONS 和 POST 估算。
11. `check-status.cmd`：输出当前哪些本地准备已完成、哪些外部上线步骤还没完成。

这些脚本不会把真实 `QWEN_API_KEY` 写进代码，也不会替用户绕过 GitHub、Netlify 或微信后台登录。它们的作用是把容易输错的命令和验收步骤收束成可重复点击的流程。

## 当前 API 接入状态

当前网站已经接入 AI 估算 API 的服务端通道：

```text
/api/estimate-drink
```

它会读取 Netlify 或本地服务器环境变量中的：

```text
MILKTEA_AI_ENABLED
QWEN_API_BASE_URL
QWEN_MODEL
QWEN_API_KEY
QWEN_API_TIMEOUT_MS
MILKTEA_ALLOWED_ORIGINS
```

如果这些环境变量配置正确，线上网站和微信小程序都可以通过同一个接口拿到 AI 估算结果。如果接口不可用，前端会使用本地估算规则作为兜底。

## 数据与隐私

当前版本默认不需要注册登录。用户记录优先存在浏览器本地：

```text
milkTeaAppData
```

AI 估算时会把饮品相关信息发送到服务端接口，再由服务端请求 QWEN。项目不应该把用户隐私信息、手机号、真实姓名等发送给 AI。当前饮品估算只需要饮品信息。

## 可上线形态

项目现在适合三种上线形态：

1. 网页版：通过 Netlify 获取公共链接。
2. 微信小程序版：迁移 UI 后，请求同一个 Netlify API。
3. iOS 版：保留规则层和数据模型，替换为原生 UI 和本地存储。

最推荐先上线网页版，因为它最快能得到公共链接，也能作为微信小程序 API 后端。

## 项目亮点总结

`今天不喝` 的亮点不是单个功能，而是几个体验组合在一起：

1. 用虚拟下单承接想喝奶茶的冲动。
2. 用 AI 把饮品变成价格、热量、糖分三个可感知数字。
3. 用 Body Score 把长期选择变成趋势。
4. 用会变化体型的小人把数据变成视觉反馈。
5. 用手动修正保证 AI 结果可控。
6. 用本地保存让用户无需登录即可开始。
7. 用服务端 API 保护密钥，为微信小程序上线打基础。
8. 用柔和 UI 和微交互降低记录压力。

它更像一个“温柔地帮你少喝一点”的小工具，而不是一个严厉监督你的健康软件。
